#微軟實習故事：程式碼、西雅圖與未來的形狀

2026 年的夏天，台北微軟辦公室的玻璃帷幕正反射著耀眼的烈日。陳宇軒（Ethan）站在 19 樓的落地窗前，手裡握著那張剛發下來、印有經典四色微軟標誌的實習生識別證。作為台灣大學資訊工程研究所一年級的學生，他歷經了三輪嚴苛的技術面試與一場關卡重重的行為面試，才終於從數千名申請者中脫穎而出，成為今年「微軟未來生涯體驗計劃（FY26 Intern Program）」的雲端架構部研發實習生。他的心中既充滿了對科技巨擘的憧憬，也夾雜著一絲對於未知挑戰的焦慮。

帶領宇軒的導師（Mentor）是一位名叫張菡文（Hannah）的高級雲端架構師。菡文在微軟已經服務了八年，眼神裡透著精明與幹練，但笑起來卻十分溫暖。第一天報到時，菡文沒有讓宇軒去看那些枯燥的員工守則，而是直接將他帶到了一間名為「Ada Lovelace」的會議室。菡文在白板上寫下了一個代號——「藍天專案（Project AzureSky）」。這就是宇軒未來兩個月實習的核心任務：利用微軟最新的 Azure OpenAI 服務與分散式向量資料庫，為亞太區的大型金融客戶構建一套具備高併發、低延遲特性的企業級 RAG 知識管理系統。

「宇軒，這個專案不是實驗室裡的玩具。」菡文看著他，語氣嚴肅而專注。「客戶的原始文件高達數百萬筆，包含複雜的法律條文和財務報表。目前的開源解決方案在處理長文本時，召回率（Recall Rate）只有不到百分之七十，且常常出現幻覺。你的任務是在八週內，將檢索精準度提升至百分之九十以上，並且必須在微軟全球實習生黑客松（Global Intern Hackathon）上進行成果展示。這會是你的終期考核。」

回到座位後，宇軒打開了微軟為他配備的 Surface Laptop Studio。看著空白的 VS Code 視窗，他深吸了一口氣。他知道，要解決長文本財務報表的檢索問題，傳統的簡單切塊（Chunking）絕對行不通。那些表格數據和跨頁面的關聯資訊，在切塊後會徹底失去上下文。他必須設計出一套創新的雙路檢索機制（Two-Way Retrieval）：結合密集向量檢索（Dense Retrieval）與稀疏關鍵字檢索（BM25），並在最上層加上重排模型（Reranker）。

實習的第一週，宇軒幾乎泡在微軟的文獻庫和 GitHub 企業版中。他發現，金融文件的最大痛點在於「年報中的表格」。當一個表格被暴力切分成好幾段時，數字就會失去欄位標籤的意義。為了解決這個問題，他決定引入 LayoutLMv3 模型，先對 PDF 進行版面分析，將表格轉化為結構化的 Markdown 格式，再將每一行數據附帶上表頭資訊，最後才送入 Azure OpenAI 的 text-embedding-3-large 模型進行向量化。這個創新的想法讓他在第一週的 Weekly Sync 會議上得到了菡文的肯定。

然而，現實的開發過程從來不會是一帆風順的。到了第三週，當宇軒將處理好的 50 萬筆測試數據導入 Azure Cosmos DB for MongoDB 的向量索引時，系統崩潰了。終端機上彈出了一連串血紅色的錯誤訊息：「Request rate is large (ExceededTimeLimit)」。由於大量的並行寫入需求，觸發了資料庫的吞吐量限制（RU/s 限制）。此時距離中期進度審查（Mid-term Review）只剩下不到四十八小時，宇軒的額頭滲出了冷汗。

就在宇軒一籌莫展、準備在辦公室熬夜時，同期的另一位數據科學實習生、來自清華大學的林佳妤（Chloe）走進了研發區。佳妤負責的是另一個負責用戶行為預測的 AI 專案，對大數據調優非常在行。看到宇軒盯著螢幕發呆，佳妤放下一杯冰美式咖啡，笑著問道：「怎麼了？被 Azure 的 RU 限制卡住了嗎？」宇軒點點頭，無奈地指著螢幕上的錯誤代碼。

佳妤拉了一張椅子坐下，敲擊了幾下鍵盤，調出了 Azure Monitor 的監控圖表。「你看，你的寫入邏輯是用單線程的 for 迴圈一筆一筆送，這會導致瞬間流量過大。你應該使用非同步並行（Asyncio），並且加上指數型退避重試機制（Exponential Backoff）。另外，把 Cosmos DB 的吞吐量臨時調高到兩萬 RU，等資料灌完再調降，這樣既能解決問題又不會超支預算。」佳妤的建議如同一場及時雨，瞬間點醒了宇軒。

在佳妤的協助下，宇軒重寫了資料清洗與導入的腳本。當他按下執行鍵，看著終端機上的進度條以每秒數千筆的速度飛速前進，且再也沒有跳出任何錯誤時，他長長地鬆了一口氣。兩天后的中期審查會議上，宇軒展示了這套高效的資料流水線（Data Pipeline）。部門協理（Director）在聽完簡報後，讚許地說道：「能在這麼短時間內解決大規模資料分散式寫入的架構問題，這已經達到正式員工的水準了。」

中期審查通過後，專案進入了最核心的調優階段。雖然資料順利導入，但初步的評估結果卻給宇軒澆了一盆冷水。在針對 1000 道金融測試問題的評測中，系統的 Hits@5（前五個檢索結果包含正確答案的比例）只有 74.5%，距離菡文要求的 90% 還有很大的差距。最嚴重的問題出在「跨指標對比」的問題上，例如當用戶詢問「比較 2024 年與 2025 年第三季度的研發投入差距」時，系統往往只能檢索到其中一年的資料。

為了解決這個痛點，宇軒決定在檢索前加入一步「查詢改寫（Query Rewriting）」。他利用 GPT-4o 作為意圖解析引擎，當使用者輸入一個複雜的比較型問題時，系統會自動將其拆解為兩個子查詢（Sub-Queries），分別去檢索不同年份的資料，最後再將檢索到的上下文進行合併（Context Fusion）。為了驗證這個想法，他連續三天留在辦公室，不斷調整 Prompt 的 Prompt Engineering 策略，甚至忘記了晚餐時間。

每當夜幕降臨，信義區的霓虹燈透過落地窗灑在辦公桌上。微軟的辦公室此時顯得格外安靜，只有工程師敲擊機械鍵盤的清脆聲響。菡文常常會在下班前走到宇軒的位子旁，遞給他一包零食，並詢問遇到了哪些瓶頸。菡文告訴他：「研發不是閉門造車。在微軟，我們講求的是 Growth Mindset（成長心態）。遇到挫折時，不要懷疑自己的能力，而是要思考還有什麼工具或架構是我們還沒嘗試過的。」這番話成為了宇軒在無數個卡關夜晚的精神支柱。

到了第五週，微軟迎來了全球實習生的年度盛事——微軟全球實習生黑客松（Global Intern Hackathon 2026）。這場比賽要求在全球各地的實習生在 72 小時內，跨團隊組隊並開發出一款具有商業價值的創新應用。宇軒、佳妤，以及來自微軟行銷部門的實習生王梓豪（Leo）組成了一個三人團隊，隊名叫作「Azure Pioneers」。他們決定將宇軒的金融 RAG 架構與佳妤的預測模型結合，打造一款名為「FinNavigator」的智能投資助理。

黑客松的 72 小時是一場體力與意志力的極致考驗。梓豪憑藉著敏銳的市場直覺，迅速設計出了極具說服力的商業企劃案與使用者介面原型；佳妤則熬夜訓練了一個輕量級的股票趨勢分類器；而宇軒則負責將他的雙路檢索與查詢改寫 RAG 引擎部署為後端 API。在比賽進行到第四十個小時時，後端伺服器突然因為連線數過高而出現記憶體洩漏（Memory Leak）。宇軒頂著熊貓眼，使用 Python 的 tracemalloc 模組逐行檢查程式碼，終於發現是一個未關閉的 HTTP 客戶端連線導致了記憶體持續攀升。

當宇軒修復完最後一個 Bug，將代碼推送到 GitHub 時，外面的天空已經泛起了魚肚白。黑客松的最後一天的 Demo Market 展位上，「Azure Pioneers」的攤位前擠滿了來自各部門的評審與工程師。梓豪用生動流暢的簡報吸引了所有人的目光，而宇軒則現場演示了「FinNavigator」如何在五秒內精準分析出一份兩百頁財報的潛在風險，並給出帶有引用來源的回答。系統表現得無懈可擊，現場響起了熱烈的掌聲。

兩天后，黑客松結果揭曉。「Azure Pioneers」在亞太區數百支參賽隊伍中，榮獲了「最佳技術創新獎（Most Innovative Technology Award）」。這個獎項不僅帶給他們實質的獎金鼓勵，更讓宇軒的 RAG 系統在整個台灣微軟高層面前留下了深刻的印象。然而，宇軒知道挑戰還沒有結束，因為黑客松的程式碼屬於原型（Prototype），要真正應用到「藍天專案」的金融客戶生產環境中，還需要經過嚴格的壓力測試與安全合規審查。

實習的第七週，是決定宇軒能否拿到正職預聘書（Return Offer）的關鍵一週。他必須對「藍天專案」進行最終的評估（Evaluation）。他使用了 RAG 評估框架 Ragas，針對真實金融客戶提供的 5000 筆測試集進行了全面的自動化評測。當評測腳本執行完畢，螢幕上跳出最終數據時，宇軒激動得差點從椅子上跳了起來：真實性（Faithfulness）達到 95.1%，答案相關性（Answer Relevance）達到 93.8%，而最關鍵的檢索召回率（Context Recall）則達到了 94.2%。他成功超越了菡文設定的 90% 目標。

除了技術指標達標外，微軟非常重視資料隱私與安全（Responsible AI）。金融客戶的資料絕對不能洩漏到公有雲的公共模型中。因此，宇軒在最後一週的優化中，專門加入了基於 Azure Presidio 的隱私資料去識別化（Data Masking）模組。在所有使用者查詢送入大模型之前，系統會自動將姓名、身分證字號、銀行帳號等敏感資訊進行遮蔽，並在回傳答案時進行還原。這個安全的設計，成為了他畢業答辯簡報中最亮眼的一頁。

實習的最後一天，台北辦公室舉行了實習生畢業成果發表會（Final Presentation）。宇軒站在大會議室的講臺上，面對著包括台灣微軟總經理、研發協理以及所有導師在內的三 layout 評審團。他用沉穩的口吻，從金融長文本的切塊痛點講起，延伸到分散式向量資料庫的寫入優化，再到雙路檢索與隱私安全的架構設計。他的簡報邏輯嚴密、數據詳實，完美的將技術深度與商業價值結合在一起。

當簡報結束，台下掌聲雷動。總經理親自站起來向他提問：「Ethan，如果未來這套系統要支持每秒數萬次的全球併發查詢，你認為目前的架構最大的瓶頸會在哪裡？你會如何優化？」宇軒自信地笑了笑，回答道：「目前的瓶頸會主要在於 LLM 的 Token 生成速度與成本。未來我們可以考慮引入快取機制（Semantic Caching），將相似度極高的常見問題解答緩存在 Redis 中，減少對大模型的直接調用；同時可以針對特定領域微調一個輕量級的開源小模型（SLM），部署在 Azure Kubernetes Service 上，以降低整體營運成本。」這個回答展現了他超越一般實習生的架構思維。

評審會結束後的下午，宇軒收到了菡文傳來的一封 Teams 訊息：「Ethan，來一下 Ada Lovelace 會議室。」當他推開門，發現裡面除了菡文，還坐著研發協理與 HR 經理。協理的手上拿著一份文件，微笑著對他說：「Ethan，這兩個月來你的表現所有人有目共睹。你不僅具備極強的動手解決問題能力，更展現了微軟最看重的團隊協作精神與成長心態。我們非常高興地代表微軟，正式向你發出 Return Offer。希望你畢業後，能以正式軟體工程師的身份，回到這裡繼續與我們一起定義未來的科技。」

接過那份沈甸甸的預聘書，宇軒心中百感交集。這兩個月的時間雖然短暫，但對他而言卻是一場全方位的蛻變。在微軟，他學到的不僅僅是 Azure 的各項雲端技術或 RAG 的架構優化技巧，更重要的是學會了如何在一流的企業環境中，與優秀的夥伴並肩作戰，將天馬行空的技術想法轉化為能夠真正改變世界的商業解決方案。

傍晚，宇軒收拾好行李，最後一次刷卡走出了微軟辦公室。回頭望去，101 大樓的燈光已經亮起。他知道，這段實習旅程雖然在這裡畫上了句點，但他作為一名科技探索者的奇幻旅程，才剛剛要開始。未來的程式碼還等著他去撰寫，而他已經準備好迎接下一個挑戰。
