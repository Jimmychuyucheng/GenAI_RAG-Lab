一、 精準檢索與細節測試（測試基礎召回能力）
這類問題用來測試系統能否在數個 Chunk 中，準確抓到特定的專案代號、人名或會議室名稱。如果 Chunking 切得不好，這些孤立的實體很容易丟失。

Q1：陳宇軒（Ethan）在微軟實習期間，導師（Mentor）是誰？他們第一天是在哪一間會議室討論專案的？

答案關鍵字：張菡文（Hannah）、Ada Lovelace 會議室。

Q2：故事中提到，由宇軒、佳妤和梓豪組成的黑客松團隊名稱是什麼？他們最終獲得了什麼獎項？

答案關鍵字：Azure Pioneers、最佳技術創新獎（Most Innovative Technology Award）。

二、 數字與指標對比測試（測試表格與數據處理能力）
RAG 系統最常在數字和百分比上翻車。這三個問題可以測試系統是否會把 A 指標的數字誤植到 B 指標，或者能否從特定段落撈出正確的小數點數據。

Q3：宇軒在第七週使用 Ragas 框架進行最終評估時，系統的「真實性（Faithfulness）」與「最關鍵的檢索召回率（Context Recall）」分別達到了多少百分比？

答案關鍵字：真實性 95.1%、檢索召回率 94.2%。（注意：可以用來檢查系統會不會跟答案相關性 93.8% 或 Hits@5 的 74.5% 混淆）。

Q4：在實習第三週時，宇軒將測試數據導入資料庫時遇到了崩潰。請問當時導入的測試數據共有多少筆？觸發的錯誤訊息完整代碼是什麼？ 

答案關鍵字：50 萬筆、Request rate is large (ExceededTimeLimit)。

三、 跨段落推理與因果關係測試（測試多步推理 Multi-hop）
這類問題的答案無法在單一句子或單一短段落中直接找到，系統必須同時檢索出「問題（痛點）」與「解法（技術）」的兩個不同 Chunk，並融合成一個回答。這非常考驗 Top-k 檢索完整度 與 LLM 的總結能力。

Q5：故事中林佳妤（Chloe）針對 Azure Cosmos DB 的寫入崩潰問題，具體提出了哪三個架構優化建議？

答案關鍵字：使用非同步並行（Asyncio）、加上指數型退避重試機制（Exponential Backoff）、臨時將吞吐量調高到兩萬 RU（灌完再調降）。

Q6：為解決金融年報中「跨指標對比（例如比較不同年份研發投入）」導致檢索不全的痛點，宇軒在架構中加入了什麼機制？該機制是如何運作的？

答案關鍵字：查詢改寫（Query Rewriting）。運作方式：利用 GPT-4o 作為意圖解析引擎，將複雜問題拆解為兩個子查詢（Sub-Queries）分別檢索，最後進行上下文合併（Context Fusion）。

Q7：總經理在畢業成果發表會上提問「如何因應全球數萬次高併發查詢與降低成本」，宇軒提出了哪兩種具體的技術解決手段？

答案關鍵字：引入快取機制（Semantic Caching / 將常見答案緩存在 Redis 中）；微調輕量級開源小模型（SLM）並部署在 Azure Kubernetes Service 上。

四、 邊界條件與抗雜訊測試（測試安全合規與拒絕回答能力）
這類問題用來測試 RAG 系統的「隱私保護能力」以及「當知識庫沒寫時，是否會瞎編（幻覺）」。

Q8：為了符合金融客戶的安全合規與隱私保護（Responsible AI），系統在使用者查詢送入大模型之前，會透過什麼工具進行什麼處理？

答案關鍵字：基於 Azure Presidio 的隱私資料去識別化（Data Masking）模組，將姓名、身分證字號、銀行帳號等敏感資訊進行遮蔽。

Q9：根據故事內容，宇軒實習所在的微軟辦公室大樓，具體位於台北市的哪一條路或哪一個行政區？

答案關鍵字：故事中僅提及「信義區」與「19 樓、可看到 101 大樓」，並未提及具體路名（如信義路或忠孝東路）。（測試系統是否會根據外部知識盲目猜測，還是會嚴格根據文本回答「信義區」）。

Q10：陳宇軒（Ethan）在微軟實習期間，每個月的實習薪資（Token / 薪水）是多少錢？

答案關鍵字：文本中完全沒有提到。（這是最經典的基準測試，正確的 RAG 應該回答：「根據提供的文本內容，並未提及陳宇軒的實習薪資。」如果系統自己編了一個數字，就代表有嚴重的幻覺問題）。
